1. EXECUTIVE ARCHITECTURAL BLUEPRINT & MONOLITH VS. MICROSERVICE DECISIONS
================================================================================

DigiDoc is designed as a decoupled client-server architecture split into:
  - Frontend: A compiled Cross-Platform Flutter application.
  - Backend: An Asynchronous Python REST API (FastAPI) backed by an ORM-mapped 
    relational database (PostgreSQL/SQLite).

--- Architectural Design Choices & Trade-offs ---

1. Why a Decoupled Backend over Monolithic Server-Side Rendering (SSR)?
   - Separation of Concerns: Flutter operates as a pure presentation layer, consuming 
     stateless JSON REST endpoints. This allows independent scaling of frontend assets 
     (e.g., hosted on global CDN edges) and compute resources (deployed on app servers).
   - Network Efficiency: Client receives pure JSON payloads instead of bloated HTML 
     documents, lowering network transit volume.
   - Native Compatibility: Paves the way for Android, iOS, and Web deployment targets 
     utilizing the exact same backend engine.

2. Monolith vs. Microservices for the API Layer:
   - Current State: Modest Monolithic API. All domains (Auth, ML Inference, Matching, 
     Booking) share the same FastAPI codebase and database interface.
   - Trade-off: Under high scaling, Monoliths suffer from "Resource Contention". 
     For example, a high-CPU CPU-bound KNN prediction blocks network threads 
     handling critical database write transactions (e.g., booking slots).
   - Production Evolution Strategy:
     * Break the system into 3 Microservices:
       a) User & Auth Service (handles Firebase token syncing, session registers).
       b) Prediction Service (houses the KNN model, runs GPU/CPU-bound calculations).
       c) Scheduling & Booking Service (highly database-optimized read-write node).
     * Communication: Services communicate asynchronously via Message Brokers (e.g., 
       Kafka or RabbitMQ) to decouple write actions, or synchronously via gRPC for 
       high-throughput internal RPC invocations.

2. THE BACKEND KERNEL: FASTAPI, ASYNCIO EVENT LOOPS, AND SQLALCHEMY ORM
================================================================================

--- FastAPI & The Asynchronous Event Loop ---

FastAPI is built on top of Starlette (ASGI toolkit) and Pydantic (data validation).
Unlike traditional synchronous WSGI web frameworks (like Django or Flask), FastAPI 
is an ASGI (Asynchronous Server Gateway Interface) framework.

1. The Single-Threaded Event Loop (asyncio):
   - Under the hood, Python utilizes an event loop (backed by uvloop, a fast C-based 
     implementation of libuv).
   - When a network request hits an asynchronous endpoint (`async def`), the loop 
     registers the socket read/write operation. If the handler encounters an `await` 
     expression (e.g., database query or file read), the event loop yields execution 
     and instantly handles another incoming request.
   - This prevents thread-blocking overhead. Traditional frameworks spawn a thread 
     per request (1MB stack memory per thread). FastAPI can handle thousands of 
     concurrent network connections on a single thread by using non-blocking I/O multiplexing.

2. Pydantic Serialization/Deserialization:
   - Every request payload is validated against a Pydantic schema class:
     * Request payload -> Deserialized into Pydantic Class -> Sanitized -> database Model.
   - This validates types, checks bounds, and prevents malformed data injections 
     before the database handler is reached.

--- SQLAlchemy ORM Engine & Session Management ---

SQLAlchemy bridges Python objects to database rows.

1. Connection Pooling:
   - Creating a database connection is highly expensive (involving TCP handshakes, 
     process allocation, and state verification).
   - SQLAlchemy's `QueuePool` maintains a pool of pre-established database connections. 
     Endpoints fetch a connection, run queries, and return the connection back to 
     the pool, preventing connection starvation.

2. Contextual Session Lifecycle (`get_db`):
   - Every HTTP transaction is wrapped in a scope-limited database Session:
     ```python
     def get_db():
         db = SessionLocal()
         try:
             yield db
         finally:
             db.close()
     ```
   - This guarantees that when an API request completes normally, changes are committed 
     or connections are safely closed, preventing database leakages and dangling transactions.

3. RELATIONAL DATABASE TOPOLOGY, NORMALIZATION, & HIGH-PERFORMANCE INDEXING
================================================================================

The database design uses third-normal-form (3NF) relational tables.

--- Database Schemas ---

1. Table: `users`
   - `id`: VARCHAR (Primary Key, matches Firebase UID/UUID)
   - `email`: VARCHAR (Unique, indexed)
   - `name`: VARCHAR
   - `role`: VARCHAR (Enum: 'patient', 'doctor')
   - `created_at`: TIMESTAMP

2. Table: `doctor_profiles`
   - `user_id`: VARCHAR (Foreign Key referencing `users(id)`, Primary Key)
   - `specialty`: VARCHAR (Indexed)
   - `latitude`: DOUBLE PRECISION
   - `longitude`: DOUBLE PRECISION
   - `is_available`: BOOLEAN
   - `availability_schedule`: VARCHAR
   - `clinic_name`: VARCHAR
   - `address`: VARCHAR
   - `phone`: VARCHAR
   - `rating`: FLOAT
   - `reviews_count`: INTEGER
   - `consultation_fee`: FLOAT

3. Table: `bookings`
   - `id`: INTEGER (Primary Key, Auto-increment)
   - `patient_id`: VARCHAR (Foreign Key referencing `users(id)`)
   - `patient_name`: VARCHAR
   - `doctor_id`: VARCHAR (Foreign Key referencing `doctor_profiles(user_id)`)
   - `symptoms`: TEXT (comma-separated list of symptoms)
   - `date`: VARCHAR (format YYYY-MM-DD, indexed)
   - `time`: VARCHAR (format HH:00, indexed)
   - `clinic_address`: TEXT
   - `additional_notes`: TEXT
   - `status`: VARCHAR (Enum: 'pending', 'confirmed', 'cancelled', default: 'pending')

--- Normalization Verification (3NF) ---
  - First Normal Form (1NF): All columns contain atomic values. No repeating groups.
  - Second Normal Form (2NF): Meets 1NF, and all non-key attributes depend entirely 
    on the primary key (No partial key dependencies).
  - Third Normal Form (3NF): Meets 2NF, and no non-key attribute has a transitive 
    dependency on the primary key (No fields depend on other non-key fields).

--- High-Performance Indexing Strategy ---

To scale lookups from milliseconds to sub-milliseconds, we apply database indexing:

1. B-Tree Indexes on Fields:
   - `users.email` (Fast login profile resolution)
   - `bookings.doctor_id` + `bookings.date` (Optimizes slot searches on a calendar pick)
   ```sql
   CREATE INDEX idx_bookings_lookup ON bookings (doctor_id, date);
   ```

2. Spatial Query Optimization (Geographic Coordinates):
   - Finding nearby doctors uses latitudes and longitudes. 
   - Traditional B-Tree indexes cannot handle multi-dimensional queries (Lat AND Lon).
   - In production, we upgrade to **PostgreSQL PostGIS** extension, wrapping coordinates 
     in a `GEOMETRY(Point, 4326)` type and building a **GIST (Generalized Search Tree) Index**:
     ```sql
     CREATE INDEX idx_doctor_location ON doctor_profiles USING GIST (ST_SetSRID(ST_MakePoint(longitude, latitude), 4326));
     ```
     This allows the query to run in log-time ($\mathcal{O}(\log N)$) using R-Tree index bounding boxes, instead of checking every row in the database.

4. MATHEMATICAL FOUNDATIONS OF THE MACHINE LEARNING PIPELINE (KNN CLASSIFICATION)
================================================================================

DigiDoc matches inputted symptoms to potential medical diagnoses.

--- The Feature Vector Space ---

  - A vector represents a patient's clinical state.
  - The feature space has **133 dimensions**, representing the 133 supported symptoms.
  - Input array:
    $$\mathbf{x} = [x_1, x_2, x_3, \dots, x_{133}]$$
    where:
    $$x_i \in \{0, 1\}$$
    - $x_i = 1$ if the symptom is selected.
    - $x_i = 0$ if the symptom is absent.

--- K-Nearest Neighbors Classifier Core ---

The classifier is instance-based, storing all training vector inputs.

1. Distance Calculation (Euclidean Distance):
   - To classify a query vector $\mathbf{x}_{\text{query}}$, we compute Euclidean distance 
     against all vectors in our training database $\mathbf{x}_j$:
     $$d(\mathbf{x}_{\text{query}}, \mathbf{x}_j) = \sqrt{\sum_{i=1}^{133} (x_{\text{query}, i} - x_{j, i})^2}$$

2. Probabilistic Class Prediction (`predict_proba()`):
   - Traditional KNN classifies based on plurality vote.
   - For healthcare context, we need probabilistic percentages.
   - We extract the $K$ closest training vectors ($K=5$).
   - The probability of a disease class $C$ is the proportion of neighbors belonging to it:
     $$P(\text{Disease} = C \mid \mathbf{x}_{\text{query}}) = \frac{1}{K} \sum_{k \in \text{Neighbors}} I(y_k == C)$$
     where $I$ is an indicator function (returns $1$ if neighbor $k$'s true disease $y_k$ matches $C$, else $0$).
   - This output is returned to the user sorted descending by probability.

--- KNN Evaluation & Analysis (Interview Preparation) ---

- Advantages:
  * Non-Parametric: No assumptions about decision boundaries.
  * Adaptive: Adding new data instantly changes inference boundaries.
  * Simple explainability: "Similar symptom profiles lead to similar diagnoses".
- Disadvantages (and mitigations):
  * **Curse of Dimensionality**: High dimensions (133) can cause Euclidean distances 
    to become equidistant, diluting accuracy.
    * *Mitigation*: Our vectors are binary and sparse. We can apply Principal Component 
      Analysis (PCA) or feature selection to reduce dimensions to key components.
  * **Inference Speed**: Comparing against every training sample takes $\mathcal{O}(N \cdot D)$ time.
    * *Mitigation*: Use spatial trees like **KD-Trees** or **Ball-Trees** to reduce 
      search complexity to $\mathcal{O}(D \log N)$.

5. ALGORITHM ENGINEERING: HAVERSINE GEOLOCATION & DYNAMIC INTERVAL MATH
================================================================================

--- Spatial Matching: The Haversine Formula ---

To find doctors near a patient without calling external Google Maps services:

Given:
  - Patient Location: $\text{Lat}_1, \text{Lon}_1$
  - Doctor Location: $\text{Lat}_2, \text{Lon}_2$
  - Earth Radius $r \approx 6371 \text{ km}$

Formula:
  - Latitude difference: $\Delta \phi = \text{Lat}_2 - \text{Lat}_1$
  - Longitude difference: $\Delta \lambda = \text{Lon}_2 - \text{Lon}_1$
  
  $$\theta = \sin^2\left(\frac{\Delta \phi}{2}\right) + \cos(\text{Lat}_1)\cos(\text{Lat}_2)\sin^2\left(\frac{\Delta \lambda}{2}\right)$$
  
  $$d = 2r \cdot \text{atan2}\left(\sqrt{\theta}, \sqrt{1 - \theta}\right)$$

In Python, we compute this across doctor records, keeping only those within our radius threshold (e.g. 15 km) and sorting the list nearest-first.

--- Scheduling: Dynamic Slot Allocator ---

To return available appointment slots, we perform subtraction on sets:

1. Masters template:
   $$\mathbf{S}_{\text{template}} = \{\text{"09:00"}, \text{"10:00"}, \text{"11:00"}, \dots, \text{"17:00"}\}$$

2. Retrieve active reservations:
   Query the database to find all bookings for a given doctor on a given date:
   $$\mathbf{B}_{\text{active}} = \{ \text{Booking.time} \mid \text{Booking.doctor\_id} = D \land \text{Booking.date} = YYYY\text{-}MM\text{-}DD \land \text{Booking.status} \ne \text{"cancelled"} \}$$

3. Output calculation:
   $$\mathbf{S}_{\text{available}} = \mathbf{S}_{\text{template}} \setminus \mathbf{B}_{\text{active}}$$
   The resulting set represents open hours returned to the client application.

6. FRONTEND COMPILATION & LOCAL SESSION CACHE TOPOLOGY
================================================================================

--- Flutter Framework Internals ---

Flutter is compiled natively down to machine code (ARM/x86/Javascript) using Dart AOT (Ahead-of-Time).

1. The Three Trees Architecture:
   - **Widget Tree**: Lightweight, immutable configuration blueprints. Rebuilt on every status change.
   - **Element Tree**: Mutable nodes managing the lifecycle of widgets and linking the widget configuration to the rendering tree.
   - **RenderObject Tree**: Heavy, mutable objects responsible for layout and paint sizing computations. Remounts only when layout layout boundaries change.

2. State Management Pipeline:
   - We utilize StatefulWidget's `setState()` to update the UI on symptom clicks and slot selections. 
   - Calling `setState()` marks the current Element node as dirty. The framework schedules a frame redraw, rebuilds the widget tree, compares changes via the Diffing algorithm, and updates the underlying RenderObject tree.

--- Local Session Cache (SharedPreferences) ---

To achieve auto-login on startup:

```
+------------------+    No Cached Session    +-----------------------+
|  App Startup     | ----------------------->|      Starter Page     |
+------------------+                         | (Role Select & Login) |
         |                                   +-----------------------+
         | Has Cached Session
         v
+------------------+    Role = 'doctor'      +-----------------------+
| Read Local Cache | ----------------------->|     Doctor Dashboard  |
+------------------+                         +-----------------------+
         |
         | Role = 'patient'
         v
+-----------------------+
|   Patient Home Page   |
+-----------------------+
```

  - High-Speed Gate: On launch, the startup thread fetches `SharedPreferences`.
  - Serialized variables are loaded into memory:
    * `AppConfig.mockRole`
    * `AppConfig.mockSpecialty`
    * `AppConfig.mockName`
    * `AppConfig.mockEmail`
  - If a session is found, routing skips the login screens and navigates directly to the dashboard, preventing network delay on launch.

7. DISTRIBUTED SYSTEMS HARDENING: RACE CONDITIONS, CACHING, AND SCALING
================================================================================

--- Mitigating High-Concurrency Booking Race Conditions ---

In high-concurrency systems, two users might read the same open slot as free and try to book it at the exact same millisecond.

Solutions:

1. Database Composite Unique Index:
   - Force database engine integrity:
     ```sql
     CREATE UNIQUE INDEX uq_doctor_slot ON bookings (doctor_id, date, time) 
     WHERE status != 'cancelled';
     ```
   - If two requests call `INSERT` simultaneously, the database forces a serialization barrier. 
     The second request triggers an integrity check error. We catch this exception 
     and show a "Slot Taken" alert in the UI.

2. Optimistic Concurrency Control (OCC):
   - Add a `version` column to the slot availability registry:
     ```sql
     UPDATE slots SET status = 'booked', version = version + 1 
     WHERE doctor_id = :id AND date = :date AND time = :time AND version = :read_version;
     ```
   - If the update fails (since another transaction updated the version), the write is rejected.

--- Scale Model serving for ML Inferences ---

- Inference Latency: running KNN predictions on CPU scales poorly with traffic.
- Solutions:
  * **Redis Caching**: Symptoms are sparse. We can hash the sorted symptom list input:
    $$\text{Key} = \text{MD5}(\text{SortedSymptomsList})$$
    Check Redis cache first. If hit, returns the diagnosis in $\mathcal{O}(1)$ time.
  * **Dedicated Inference microservices**: Move the model to a separate Python process 
    or dedicated platform (e.g. Triton Inference Server) to isolate CPU-bound tasks.

8. TECHNICAL STAR INCIDENT NARRATIVES (PRODUCTION BUG DEBUGGING STORIES)
================================================================================

--- Incident A: Dart Covariance Map Runtime Crashes ---

  - **S**ituation: Clicking a symptom in the checkbox selector list threw a red screen 
    with: `type '() => Map<String, dynamic>' is not a subtype of type '(() => Map<String, String>)?' of 'orElse'`.
  - **T**ask: Resolve the runtime generic covariance type mismatch.
  - **A**ction:
    * Identified that initializing lists of maps implicitly inferred the compiler type 
      as `List<Map<String, Object>>`.
    * Passing this into `.firstWhere` to modify a map reference threw covariant errors 
      because Dart's type system prevents writing dynamic properties into restricted maps.
    * Wrote a custom type-safe linear search helper `_findSymptom(name)`:
      ```dart
      Map<String, dynamic>? _findSymptom(String name) {
        for (var item in rawSymptomsList) {
          if (item['name'] == name) return item;
        }
        return null;
      }
      ```
  - **R**esult: Eliminated all generic type mismatch runtime errors, providing robust list updates.

--- Incident B: Physical Device Network Loopback Blockages ---

  - **S**ituation: Physical Android phone connected via USB debugging failed to connect 
    to the local FastAPI development server, throwing `Connection refused`.
  - **T**ask: Route HTTP traffic from the USB port interface back to the local computer server.
  - **A**ction:
    * Recognized that physical devices do not share loopback addresses with emulators (which use `10.0.2.2`).
    * Established ADB port forwarding:
      ```bash
      adb reverse tcp:8000 tcp:8000
      ```
    * Configured the app `baseUrl` to `http://localhost:8000`. ADB routes all requests 
      directed to localhost inside the phone back to localhost on the host computer.
  - **R**esult: Successfully bypassed device network bounds, restoring communications.
